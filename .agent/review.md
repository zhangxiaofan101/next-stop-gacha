### F8 — [P1] 11 条线路宣称的弹性天数无法通过行程单调出

线路名称去日数、卡面改为“约 min~max 天 · 默认 N 天”后，名称矛盾已消失；但 design/state 又承诺 `days` 每档“有 plan 支撑、装入行程后各站天数可调”。实际行程单的每站选项是 `城市.days ∪ 默认 stop.days`，枚举这些选项的笛卡尔积后，**11/37 条线路至少一个声明档位不可达**：河西走廊 `days=[5,7]`、默认 6 天，但四站最小合计就是 6，无法调出 5 天；北疆伊犁 `days=[7,10]` 最小 8 天，无法调出 7；同类还包括青海湖祁连、晋北古建、东北雪国、呼伦贝尔阿尔山、江南水乡、粤港澳、海南环岛、赣东北、胶东海滨。线路自身的 `plans[]` 文字并不会驱动装入后的 stop 分配或下拉选项，因而“有 plan”不能证明 UI 可兑现该档位。M25 新增 10 条线路没有增加问题数，但也没有补上构建期的可达性校验。

建议处置：→ **data/schema/code fix**。若要保留弹性档位，应为每个线路 plan 存一套可执行的 stop 天数分配，装入时让用户选择或默认某方案；构建器校验每个 `days` 值都有对应分配、分配总和正确、每站值能在行程单显示并被路书消费。若不准备实现多套分配，则删除不可达的 `days`/plan 并收窄卡面区间。不要只放宽城市下拉到任意 1..max，否则可能生成没有城市 plan 支撑的停留天数。修复后删除本项。

### F11 — [P1] 跨省目的地让“点亮省份数”与地图实际点亮数不一致

`renderMap()` 有意用省短名 `includes` 判断填色，因此打卡泸沽湖（`province="云南·四川交界"`）会同时点亮云南和四川；但地图统计条和 header 足迹胶囊都对原始 `province` 字符串做 `Set`，把该值只算作一个“省份”。例如只打卡泸沽湖时，地图亮 2 省却显示“点亮 1 个省份”；先打卡杭州再打卡泸沽湖时，地图亮浙江/云南/四川 3 省，却显示 2 省。state 中“泸沽湖后云南+四川 fill 且统计=footPill=2 省”的验收只证明两个错误计数彼此一致，没有验证计数与实际填色一致，且直接违背 goal/M10 文案“点亮 M 个省份”的用户语义。

建议处置：→ **code + regression test**。抽出唯一的 `litProvinceNames(visited)`（以 `CN_MAP.prov[].n` 为规范集合，沿用当前 `includes` 规则），地图填色、地图统计条和足迹胶囊全部消费同一个集合；至少加入“仅泸沽湖 = 2 省”“杭州 + 泸沽湖 = 3 省”的回归断言。修复后删除本项。

### F12 — [P1] Open-Meteo 展示缺少 CC BY 4.0 要求的许可链接与改编说明

页面目前仅显示纯文本“Open-Meteo 估算”，没有可点击的数据来源/许可链接，也没有说明温度被四舍五入、天气码被转换为中文与 emoji。Open-Meteo 官方 [License](https://open-meteo.com/en/license) 明确要求 appropriate credit、许可链接以及在有修改时注明；当前实现对数据做了可见转换，却只写名称，不足以满足其 CC BY 4.0 署名条件。另外，当前调用的免费 API 端点按官方 [Terms](https://open-meteo.com/en/terms) 仅供非商业使用；仓库的公开发布约束没有记录这一部署前提，后续若站点商业化会无声越界。

建议处置：→ **UI attribution + design/deployment constraint**。在天气卡可见位置加入指向 Open-Meteo 的来源链接和 CC BY 4.0 许可链接，并用简短文案注明“经本站整理/转换”；打印/文本导出若携带天气数据，也应保留紧邻署名。把免费端点“仅非商业部署”的限制写入 design 或发布检查；如需商业用途，切换到符合条款的商业方案。修复后删除本项。

### F13 — [P2] visited 的持久化清洗把合法线路 id 当成合法城市 id

M10/state 定义 `visited` 只存城市 id，线路详情也刻意不提供打卡按钮；但 `loadLS()` 用全部 `DATA` id 清洗 `visited`。因此只要旧数据、手工修改或跨版本状态中出现任一合法线路 id，它就会永久保留，并被 `state.visited.length` 算作“去过的目的地”；`renderMap()` 也会把线路记录纳入省份统计和填色，尽管代码注释声称这里“只存城市 id”。现有损坏 localStorage 验收只用了不存在的垃圾 id，没有覆盖“schema 内合法、业务上非法”的 route id。

建议处置：→ **code + persistence regression test**。为 `visited` 单独使用 `new Set(DATA.filter(d => !d.stops).map(d => d.id))` 清洗，并顺手去重；加入混合输入 `[城市 id, 线路 id, 不存在 id, 重复城市 id]` 的加载测试，断言最终只剩唯一城市 id。修复后删除本项。
