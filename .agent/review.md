### F18 — [P1] 旧版本已持久化的独库行程没有迁移，升级后仍输出原来的错误路书

新 `addRouteToTrip()` 已把 `r` 从布尔标记改为线路 id，新加入的独库默认 5 天行程能正确选中三段 `leg`。但旧版本把所有线路 stop 持久化为 `r:1`（`4d8e494:index.html:1764`），当时独库还会写入 `yili:2 / bayanbulak:1 / kuqa:2`。当前 `loadLS()` 只校验城市 id/天数，原样保留 `r:1`（`index.html:1366-1378`）；`planForStop()` 用 `byId(d.rid)` 回查线路，`byId(1)` 必然失败并回退城市 plan（`:2157-2165`）。因此真实旧用户升级后仍保留伊犁/巴音布鲁克/库车旧三站，继续看到“伊犁 2 天按 5 天版改编 + 巴音布鲁克 1 天按 2 天版改编”；state 中的“旧 `r:1` 持久化不崩”只证明不抛异常，没有修复 F18 对已有用户的原始故障。

建议处置：→ **persistence migration + regression**。在 `loadLS()` 中对旧 `r:1` 做可识别迁移：至少覆盖旧独库三站组合（包括原顺序与用户可能的重排），迁移到新 `tianshan-tianchi / bayanbulak / kuqa` 默认分配并写入 `r:"route-duku-highway"`；无法安全识别的旧线路标记至少要明确降级，不要默默声称已修复。回归用此次发布前真实 localStorage payload 启动页面，断言路书无“按 5 天版改编”且包含哈希勒根/乔尔玛驾驶段。

### F21 — [P1] `leg` 只按单站天数判定生效，线路被重排/删站/跳过已有站后仍会输出固定顺序文案

`planForStop()` 只检查当前 stop 的 `rid`、id 和天数，不检查同一线路的 stop 是否齐全、连续且仍按 `route.stops` 顺序（`index.html:2157-2165`）。但行程单明确允许上移/下移/删除，顺路排序也会直接重排 `state.trip`（`:2029-2045, 2088-2093, 2454-2460`）；`addRouteToTrip()` 遇到已在行程的同 id 站点还会直接跳过，不会为已有条目补 `rid`（`:1761-1765`）。复现：默认装入独库后把“巴音布鲁克”上移到第一站，它的路书仍输出“D1 乌鲁木齐经奎屯到独山子…”，同时逐站交通标题却是“上海 → 巴音布鲁克”；随后的“乌鲁木齐·天山天池”又仍写“次日一早西行独山子上独库”。删掉中间站、行程已有巴音布鲁克后再点“整条装入”也会产生同类的半条线路+全量分段文案。

建议处置：→ **all-or-nothing route context + interaction regression**。`leg` 应在路线组级判定：同 `rid` 的全部 stop 必须齐全、连续、顺序与默认天数都匹配才整组启用 `leg`；任一条件失效就整组回退，不能按单站混用。或者在用户重排/删站时明确解绑整组 `rid`。回归覆盖上移、删中站、已有重复站、顺路排序和 6 站上限只装入部分线路。

### F22 — [P1] 独库明确翻越 3000m+ 垭口却仍是 `alt:false`，“避开高海拔”会错误保留该线路

`route-duku-highway.seasonNote` 明写“哈希勒根等垭口 3000m+”，方案和亮点也要求翻越该垭口（`data/routes.json:301,328-340`），但该卡仍是 `alt:false`（`:356`）。前端“避开高海拔”的唯一判据是 `if (noAlt && d.alt) return false`（`index.html:1533-1542`），因此用户明确排除高海拔时仍会看到独库，详情也不会显示高原反应警示。构建器只校验“任一 stop 城市 `alt:true` 则线路必须 true”（`tools/build.py:152-154`）；本次为线路选的三个代理 stop 都是 `alt:false`，所以无法捕获“途中穿越高垭口”这种线路自身的海拔风险。

建议处置：→ **data fix + filter regression**。将独库线路卡改为 `alt:true`，并断言打开“避开高海拔”后 `route-duku-highway` 不进入结果/扭蛋池、详情展示高海拔警示。构建器不宜从自由文本猜海拔，但可为独库这类已知的线路级风险增加定向数据断言或审计用例。
