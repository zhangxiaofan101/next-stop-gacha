### F47 — [P1] fail-open 消除了正常同步的 500，但也让限流在真实 KV 写频约束下完全放行

`checkRateLimit()` 现在捕获 KV 的 `get/put` 异常并返回允许，确实修掉了 GET→PUT 的 500；PUT 也已把限流移到同步码存在性读取之前。但计数仍以 `get(counter) → put(counter+1)` 反复改写同一个日 key，而 Workers KV 官方限制同 key 每秒最多写 1 次。第二个高频请求开始，计数写失败即 fail-open；本轮按该约束连续重放 100 次同 IP、同 bucket 的未知码 PUT，100 次全部穿过限流并执行昂贵的同步码读取，均为 404、无一次 429。并发 read-modify-write 还会丢计数，因此这不是仅精度稍差，而是攻击者能持续禁用防线、消耗 KV 免费读额度。

新增测试只断言“限流存储失败时不 500”；所谓“未知码 PUT 仍被限流”仍使用无写频约束的 `Map`，没有验证真实故障路径。需改用不会依赖 KV 高频同 key 写的防滥用机制，例如 Workers Rate Limiting binding 作突发保护，或由 Durable Object/等价协调者串行计数；回归必须在计数存储拒绝高频写或并发时仍能观察到拒绝，而不能把 fail-open 本身当成限流通过。若保留宽松降级，也应有独立、实际可工作的第一层配额保护。

### F48 — [P1] 服务端并集只保护已可见的顺序写；并发或陈旧读仍会丢更新

PUT 现在会在服务端读取现值、对 `favs/visited` 做并集再写回，且客户端会吸收响应中的合并快照；这修复了“服务端已看见前一次写入”的顺序场景，也继续保持 `trip/cmp` 不变。但 KV 不是同 key 的原子 read-modify-write：若 A/B 同时读到 R，两边仍分别写 `R∪A`、`R∪B`，最后一个 KV 写覆盖另一个。本轮用 barrier 让两个 handler 在写前都读完，两个请求都返回成功及各自并集，最终存储却只剩 B，原竞态未消失。

代码注释把残余窗口描述为“毫秒级”，但 KV 跨 location 的陈旧读可持续 60 秒或更久；实现者的生产复验是顺序 PUT，不是 F48 的并发复现。只要被覆盖的设备在下次补传前丢失/清空，用户已看到成功 toast 的记录仍会从云端消失。需把每个同步码的“读→并集→写”放在单一强一致协调者中（例如一 code 一 Durable Object），或改成天然可合并的存储模型；加入两个真正并发写者的回归，并断言最终权威快照包含全集。若接受服务端并集为持久机制，还须把 design.md 当前“PUT 全量覆盖”同步为实际语义。
