### F14 — [P1] M28 只封住新入口，旧版本已持久化的线路 id 仍作为单站留在行程

M28 已让新彩蛋候选排除线路卡，`insertOnWay()` 也拒绝线路 id；但 `loadLS()` 仍用全部 `DATA` id 清洗 `state.trip`（`index.html:1369-1372`），不像 `visited` 那样限定城市 id。旧版本的顺路彩蛋曾能把线路 id 直接写进 `nextstop_v2.trip`，所以升级后的真实用户可能已经带着 `{id:"route-duku-highway", days:5}`。当前加载会保留它，行程/路书继续把整张线路当一座城市，而不是展开 `stops`。

建议处置：→ **code + persistence regression test**。最低风险方案是在 `loadLS()` 中用城市 id 集合清洗 `trip`，并校验条目对象与天数；更复杂的“自动把旧线路迁移成 stops”需另行定义 6 站上限、重复站和默认天数冲突。至少加入 `[城市, 线路, 不存在 id]` 只保留城市的升级回归。

### F15 — [P2] M30 的自驾文案批量仍留 4 张卡没有明确的上海自驾口径

M30 前，江浙沪 42 张卡中有 34 张 `transit` 不含“自驾”；本提交补了 30 张，当前仍剩 `舟山·普陀山`（`data/data-a.json:523`）、`温岭·石塘`（`:743`）、`丽水·松阳`（`:856`）、`盐城·条子泥`（`:1493`）。四条都有泛称“车程”或落地接驳，但后三条仍只写“高铁后再车程”，没有用户从上海全程自驾的时间；普陀山也没有按 design 所述海岛特例明确“自驾至码头 + 船”。因此 state 中“42 条中 30 条缺自驾并已逐条补”的计数口径把这 4 条泛称车程视为已覆盖，但用户实际拿不到明确自驾方案。

建议处置：→ **data fix + count assertion**。为四条补上海自驾全程口径；普陀山写到沈家门/朱家尖码头再乘船。构建或一次性审计断言江浙沪城市卡 `transit` 均含明确“自驾”或被列入有理由的例外清单。

### F16 — [P2] M26 JSON 导入对合法 JSON 的非对象值抛未捕获异常

`importJSON()` 只捕获 `JSON.parse` 失败，随后直接访问 `obj.favs` / `obj.visited`（`index.html:2382-2386`）。粘贴合法 JSON `null` 后点击“合并导入”，解析成功但在 `obj.favs` 处抛 `TypeError`；数字等非对象值也不符合导入 schema，却没有一致的格式错误提示。用户看到的是按钮无反馈，控制台报错。

建议处置：→ **code + input regression test**。解析后先校验 `obj && typeof obj === "object" && !Array.isArray(obj)`，不合格走“格式不对/没找到可导入记录”提示；覆盖 `null`、数字、数组、字段非数组和正常对象。

### F17 — [P2] M28 用已取整距离计算绕路增量，会显示负绕路公里数

`bestInsertion()` 用 `hav()` 的整数公里结果做三边差（`index.html:2119-2126`），取整会破坏三角不等式；UI 又原样显示 `+绕${x.add}km`（`:2113-2114`）。全量 249 城枚举的一站行程回归出现 45 个负增量配对；例如行程只有「温岭·石塘」时，候选「宁波」为第一名并显示“+绕-1km”。这不是理论边角：负值会优先排序到前三名直接露出。

建议处置：→ **code + geometry regression test**。绕路计算使用不取整的 haversine 距离，最终展示时再 `Math.max(0, Math.round(add))`；至少断言任意候选 `add >= 0`，并覆盖上述石塘→宁波用例。
