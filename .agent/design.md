# Design — 下一站，去哪玩

## 架构（三层，全部封装在单文件页面 + 数据目录中）

```
┌─────────────────────────────────────────────────────┐
│ 展示层  浅色卡通视觉系统 · 筛选控制台 · 卡片网格      │
│         详情页 · 对比表 · 扭蛋机 · 行程单 · 路书视图  │
├─────────────────────────────────────────────────────┤
│ 决策层  过滤/排序引擎 · 条件随机 · 顺路排序(贪心)     │
│         距离→交通方式启发式 · 预算估算 · 路书装配     │
├─────────────────────────────────────────────────────┤
│ 数据层  六区 JSON + 补丁 JSON（仓库 data/）           │
│         构建脚本：合并 → schema 校验 → 注入页面常量   │
└─────────────────────────────────────────────────────┘
```

部署层不改变产品的单文件零依赖约束：构建仅把 `index.html` 复制到隔离产物目录，边缘 Worker 剥离 `/next-stop-gacha` 前缀后交给静态资源服务。Cloudflare 路径 Route 比 Lab 的整域 Custom Domain 更具体，因此游戏仓库可以独立发布，同时 Lab 首页继续由 Lab 仓库负责。

## 信息组织

| 数据 | 来源 | 更新频率 | 存储 | 用途 |
|------|------|---------|------|------|
| 目的地主数据（144 条） | 模型知识分区生成 + 枚举校验 | 手工低频 | data/六区 JSON | 筛选/展示/随机 |
| 补丁（坐标/住宿/当地交通/体力档/高海拔） | 同上，独立补丁文件 | 手工低频 | data/补丁 JSON | 行程规划/路书/体力筛选 |
| 用户状态（收藏/对比/行程） | 用户操作 | 实时 | localStorage | 跨会话保留 |

## 决策机制

- **过滤**：按属性语义分两类（2026-07-13 用户拍板，判据＝低档位是「主动偏好」还是「被容忍的约束」）
  - **偏好型 → 多选 OR**（选中取并集，选一个不牵连另一个）：地区、季节、冷热、**体力**（躺平/正常/费腿/硬核）——想小众/想硬核是主动选择，不该被自动塞入不想要的档
  - **容忍型 → 天花板**（点一档＝自动含所有更省/更易达的低档，chip 填充式点亮）：**花费**（点档＝预算上限，含更便宜的）、**抵达难度**（点档＝可达性上限，含更好到达的）——高容忍必然接受低，单调
  - 玩法标签为 AND（收窄）；天数为桶映射（2/3/4-5/7/10+ 与目的地 days 数组求交）；关键词全字段模糊
  - 高海拔(>2500m) 为独立二元排除/警示开关，与体力四档正交
  - 实现：偏好型与容忍型共用 `state[key]` Set 与同一条 `!set.has(d.value)` 过滤；差异只在 chip 点击——容忍型点第 t 档填充 {0..t}，故 `.has` 天然得到"≤天花板"
- **空池治理**（M21，2026-07-13 用户拍板，AND 收窄语义保留）：两道防线——①事前：每个未选 chip 实时标注"点下去还剩几个"（按各组真实点击语义模拟），0 的置灰但仍可点；②事后：结果为 0 时计算去掉哪个单一条件（单个玩法 tag / 整组 / 搜索词 / 只看收藏）能救回最多，按救回数给一键定向放宽（保留其余条件），扭蛋空池同样给最优放宽+一键再扭；「全部清空」降级为兜底
- **属性打标原则**（2026-07-13 用户拍板，适用 M14 起所有新属性）：不强制单值——地方有几副面孔就打几档（多值数组），没有明显该属性个性的**留空=通配**（任何该维筛选都命中）；不硬拍主观档位。M16 抵达难度除外（地理事实，天然单值）
- **条件随机**：在当前过滤结果内均匀抽样；动画减速后揭晓
- **顺路排序**：以上海为起点的最近邻贪心，用户可手动覆盖
- **顺路彩蛋（M28 修订语义）**：候选=未在行程的**城市记录**（线路卡排除——展开装入是行程单动作，不走彩蛋通道）；「顺路」按**最小绕路插入**度量：对每个候选算插入现有环线（上海→各站→上海）任一段的最小增量里程，取增量最小的前 3 个展示「+绕 XXkm」，点击插入最优位置而非追加末尾。与主线的衔接由此保证（2026-07-15 用户反馈「彩蛋跟主线衔接不大对」）
- **交通启发式**：按坐标球面距离分档（短途打车/自驾 → 中程高铁 → 长程飞机），西藏/新疆/海南/港澳强制倾向飞机+包车；**leg 陆路段例外（F29，2026-07-16）**：相邻两站同属一条完整启用的 leg 组（`legEligibleIndices`+同 rid 判定）时，`legInfo` 收 `overland=true`，60km 以上一律「包车/自驾」不判飞机——招牌自驾/穿越线（G318/阿里南线/塔里木沙漠公路）两站再远也是陆路翻山穿沙、无航班，飞机判定与 leg 陆路叙事矛盾且踩「编造」红线；中途过夜由 `leg.stays` 承载，站间显示累计陆路里程/时长。只作用于站间；上海往返大交通仍走飞机。破坏整组（F21）后 overland 随之失效、回退通用距离判定。**显式交通语义（F30，2026-07-16）**：`leg` 只承诺逐日文案与住宿、不代表陆路，故非公路段（三峡游轮）用 `stop.leg.transport`（枚举 游轮/轮渡/火车，build 校验）直接指定，`legInfo` 显式模式优先于距离/overland 启发式（游轮 🚢≈26km/h）——避免 F29 的 overland 把游轮误标自驾；`tripLegs` 从「进入该站」的目标站 leg 读 transport。**进出门户（F31，2026-07-16）**：线路可选 `entry`/`exit`=门户城市 id（如三亚进、海口出，与首末停留站不同），`tripGateway()` 仅在行程恰为某线路完整整组时生效，改写首末大交通端点坐标与标注——行程单/逐日速览/详情/预算四处统一用真实进出点，门户→首站、末站→门户的岛内短驳由 leg 文案承载；破坏整组后门户同 overland 一起失效、回退首末停留站。渤海/黄海跨海修正按方向生效——胶东三市/大连与真跨海对岸（南方沿海、隔海峡的东北或华北北缘）≥500km 判飞，同侧陆路走廊（青岛→京豫晋、大连→东北）不强制；**江浙沪自驾修正（M30）**：两端都在江浙沪（上海视为江浙沪）的段，160~400km 档也标「高铁/自驾」（用户自驾，2026-07-15）——海岛登船段不受此影响（模式文案不改变轮渡事实，transit 文案负责细节）；输出明示为估算
- **预算估算**：每站消费档位×天数得住宿餐饮区间，加大交通按距离粗算，输出人均区间
- **路书装配**：按每站所选天数匹配（或就近降配）该站行程方案，串接站间交通段，输出逐日视图 + 纯文本导出 + 打印样式。**线路视角逐日文案（F18，2026-07-16）**：线路卡 stop 可带可选 `leg={route, stays}`——「这一段路怎么开、当晚睡哪」的线路视角文案；路书对该 stop 优先用 leg 而非城市独立游 plan（城市 plan 描述的是「住这座城放射游」，与自驾线路语义不符）。leg 是「线路上下文」文案（写死前后站顺序），**整组判定**（F21）：同 `r` 的全部 stop 必须齐全、连续、同序、天数=默认，才整组启用 leg；用户上移/删站/顺路排序/只装半条 → 组失效整组回退城市 pickPlan（不能半条线路配全量分段文案）；trip 条目记 `r=线路 id` 供路书回查。旧版 `r:1`（线路 id 已丢）loadLS 降级迁移为独立城市站+天数夹合法档（F18）。缺 leg 的线路 stop（多数线路，途经点是真目的地城市）仍走城市 plan 就近降配，行为不变。**为何需要**：M31 把独库/环线迁成线路卡后，若 stop 指向大区域城市卡（独库→伊犁），城市 5 天区域方案被压进 2 天，路书吐出「伊犁 2天·按5天版改编」的错误内容——leg 覆盖是把线路自己的逐日叙事带进路书的正解。**逐日骨架（M29）**：路书开头给逐日速览，每天一行「日期/D 序号 · 当日动作（赶路段=从 A 到 B+交通方式，停留日=游 X）· 宿 X」；宿地来源：基地型城市=城市名（片区细节看该站住宿行），路线型（ROUTE_STAY）=方案级 `stays` 每晚落脚点数组（schema 可选字段，build 校验长度==该方案 days）；行程单可选**出发日期**（不填则维持 D1/D2 记法），填了则骨架与逐站段落都换算具体日期。速览是新增层，逐站详情段（方案/美食/住宿/市内/天气）全保留（2026-07-15 用户：关键信息要有，其他信息也要保留）
- **分享/备份（M26）**：打卡+收藏的跨设备迁移（2026-07-15 用户拍板：对象是打卡收藏记录，非行程）。三形态同源：①分享链接=URL hash 携带 favs/visited id 列表，打开时弹确认后**并集合并**进本地（绝不覆盖行程/对比，导入后清 hash）；②二维码=零依赖 QR 编码器（byte 模式+纠错 M，canvas 绘制）编码同款链接，超长（>约 1500 字符）提示改用 JSON；③JSON 导出/导入=同两字段的复制/粘贴。入口克制：工具行一个小按钮开小弹层，二维码默认收起（用户要求：别做太大太占地方）
- **卡片收录原则（M31，2026-07-15 用户拍板定稿）**：①城市卡之间只能并列，包含关系只允许线路卡→城市卡（stops 引用）；②**卡名=大众目的地语义而非行政区划**——纯名义行政包含（杭州 vs 千岛湖）不改名不算违例；③**城市卡=单城市级别，环线/多点串联玩法归线路卡**（西宁·青海湖环线与独库公路两张已迁移：前者收窄为「西宁」纯城市卡，后者删除并入 route-duku-highway）；④一张卡的行程方案不得吃掉另一张独立卡的核心内容——越界日程改域内内容+「可接《某卡》」提示；⑤同名歧义收窄命名消解（黄山→黄山风景区、武功山→萍乡·武功山）
- **实时天气（M23）**：唯一的外部实时接入，免 key 开放 API + 缓存 + 静默降级，估算口径恒标来源；核心功能对其零依赖
- **玩法节奏（基地型 vs 路线型）**：绝大多数目的地是基地型（住一处放射游）；少数记录玩法主体是沿线移动、按晚换落脚点（川西小环线、呼伦贝尔一类，现存 6 张）。路线型用小清单标注（不进数据 schema、不做筛选项），路书对应站在住宿行前给「沿线多点换宿、按每晚落脚点分段订房」提示。收录口径从严：单基地+一晚特色住宿（林芝/神农架/漠河）与双基地（乐山峨眉/晋东南）不算；**吞并其他独立卡的环线不允许以城市卡形态存在（M31 拍板，独库/青海湖环线已迁线路卡），现存 6 张的途经点均无独立卡、暂维持，若后续为其途经点建卡须同步迁移**。线路卡无需此标注——装入行程即展开为多站，换宿是天然语义

## 存储与精确性

- 必须精确：schema 枚举字段（构建时强校验，错即拒绝注入）
- 允许漂移：坐标（两位小数，仅用于估算）、住宿/交通建议（知识性文案，页面挂免责）
- 用户输入：无必填输入；全部交互为点选

## 模块列表

### M1 — 目的地数据 schema 与六区数据 [R2 · S3]
144 条核心资产；schema 设计决定所有下游功能的上限，数据错误直接伤害用户信任。

### M2 — 补丁字段（坐标/住宿/当地交通） [R1 · S2]
机械但量大；坐标精度影响交通估算，住宿信息按用户的住宿档位与位置偏好定制（具体品牌偏好不入公开文档，存 assistant 私有记忆）。

### M3 — 构建管线（合并/校验/注入） [R1 · S3]
简单脚本但是质量闸门：枚举、必填、去重、坐标范围全部在此拦截。

### M4 — 筛选/搜索/排序 [R1 · S2]
标准过滤引擎；标签 AND 语义与天数桶映射是仅有的两个非平凡点。

### M5 — 卡片/详情/对比 [R1 · S2]
信息密度与可读性的平衡；对比表复用同一数据投影。

### M6 — 扭蛋随机 [R1 · S1]
核心卖点但实现简单；动画节奏是体验关键。

### M7 — 行程规划器（顺路排序/交通启发/预算） [R2 · S2]
启发式的合理性决定"能不能真用"；估算必须明示，不可伪装精确。

### M8 — 路书生成与导出 [R2 · S2]
逐日装配逻辑 + 文本导出 + 打印 CSS 三种输出的一致性。

### M9 — 视觉系统（浅色卡通） [R2 · S1]
用户明确审美要求；CSP 下无外部字体/图片，卡通感全靠 CSS 造型。

## 二期扩展（2026-07-13 排期，细节见 state.md）

**属性扩展机制**：新属性一律走「schema 增字段 → 补丁文件补数据 → 构建校验收紧枚举 → 前端加筛选组」四步，与既有管线同构；带 🌫️ 的模块先与用户拍板档位定义再动数据。

### M10 — 去过打卡 + 足迹统计 [R1 · S1]
去过语义只挂城市记录（线路卡不打卡、不计入统计——去过按城市算）。visited 数组入 localStorage 主 state，与 favs 同套持久化/非法 id 过滤。三处入口：详情页打卡按钮（已打卡转弱化样式）、卡片「✓ 去过」角标、筛选区「隐藏去过的」开关——开关走 onlyFav/noAlt 同一套覆盖机制，自动进 chip 计数、空池定向放宽与扭蛋池摘要。足迹统计=「去过 N 个目的地 · 点亮 M 个省份」，N>0 才显示（header 胶囊），省份按记录 province 字符串去重；M24 足迹地图复用同一份 visited。

### M11 — 海外版数据 [R2 · S2]
复用 schema，换区域枚举与出发地假设。

### M12 — 住宿文案去品牌化 [R1 · S2]
公开页面不暴露个人品牌偏好；hotel 字段改档位化+位置化表达，页面模板同步。

### M13 — 花费筛选 [R1 · S1]
cost 字段已存在，纯前端加一组筛选 chips，零数据成本。2026-07-13 语义定为**容忍型·天花板**：点一档自动含所有更省的低档（chip 填充式点亮），过滤 `.has` 不变、仅改 chip 点击填充逻辑。

### M14 — 体力消耗属性 [R2 · S2]
新增 effort 四档（躺平/正常/费腿/硬核）+ 高海拔（>2500m）独立标记（2026-07-13 用户拍板）；按目的地代表性玩法打分，批量补数据。筛选语义＝**偏好型·多选 OR**（选中取并集，不自动联动）；打标＝**多值数组，空=怎么玩都行（通配）**；高海拔为独立二元排除/警示开关。筛选组挂进 M21 的计数/放宽机制。

### M15 — 目的地扩容二批 [R2 · S3]
144 → 218（二批实际 +74，2026-07-13 交付）；复用分区生成+补丁+校验管线，重点补江浙沪县域/海岛、晋豫古建、云贵小众、帕米尔、东北冬季。去重后 74 条为「真实+知名度够+不与既有重复」的可靠上限，未凑数补至 220+。

### M16 — 抵达难度 [R1 · S2]
直达 / 一次中转 / 折腾 三档（2026-07-13 用户确认要做、同日交付），从 transit 半自动派生（启发式：显式"转/换乘/再X"计一跳；未写转字但另有一段限定耗时>30min或坐船/索道/包车进山的落地段也计一跳；近郊自驾算直达，远途飞机后自驾不折算；边境证件/偏远海岛小船/全程约1天等升级为折腾）+ 人工逐条复核全 218 条，影响"周末能不能去"判断。筛选语义＝**容忍型·天花板**（点一档含所有更易达的低档，复用 M13 的 CEIL_GROUPS 机制）。

### M17 — 同行适配 [R1 · S2]

### M19 — 对比/行程入口显性化 [R1 · S2]

### M18 — 多城联游线路卡 [R2 · S2]
解决"筛 1 周扭蛋空池"（2026-07-13 用户提出，呈现形式拍板：线路卡混入主池）。

**schema（implementer 拍定，非数据层面模糊点）**：独立文件 `data/routes.json`（不改动六区城市文件，零迁移成本），记录字段与城市记录同构（id/name/emoji/province/crowd/cost/seasons/seasonNote/days/transit/tagline/tags/food/museums/architecture/highlights/plans/coords/hotel/local/effort/alt/difficulty 全部必填，直接复用既有校验与筛选逻辑），新增两个字段：
- `regions`: string[]，途经的全部大区（筛选用，多选 OR 命中任一）；`region` 字段保留为主要/起点大区（卡片徽章展示用）。城市记录不填 `regions`，运行时用 `d.regions || [d.region]` 兜底，matchOne 的地区判断因此对两种记录统一处理，城市记录零改动。
- `stops`: `{id, days, leg?}[]`，长度 2~4，引用既有城市记录的 id、给出建议天数，即游览顺序。这是"整条装进行程单"的payload——点击即把 stops 逐条 push 进 `state.trip`（带 `r=线路 id`，遵守既有 6 站上限，超出部分提示部分装入），后续顺路排序/交通估算复用现有管线；路书对带 `leg={route,stays}` 的 stop 用线路视角逐日文案（F18，见「路书装配」），无 leg 的 stop 仍走城市 plan 就近降配。`leg.stays` 须与 stop.days 等长（build 校验）。

`stops` 字段的存在与否是判别"这是线路卡还是城市"的唯一标志（不新增 type 字段，避免侵入城市 schema）。build.py 对含 `stops` 的记录额外校验：长度 2~4、每个 id 必须存在于合并后的城市池、`regions` 非空且 ⊆ REGIONS。

**天数与高海拔契约**（2026-07-14 codex F3/F4 后固化，构建器强制；2026-07-15 codex F8 用户拍板补全）：站点天数分配是唯一事实真相=默认装入分配；`days` 枚举是**有对应 plan 文字支撑的弹性档位**（每档必有 plan；短档=压站玩法，参考对应天数版方案文字），作天数桶筛选，须满足 min(days) ≤ Σstop.days ≤ max(days)；线路名称不带日数（构建器强制），卡面/详情展示「约min~max天 · 默认Σ天」；stop 建议天数 ∈ [1, 该城市 days 上限]（途经短停允许低于城市独立游枚举）；**线路装入行程单的站点下拉放开为 1~城市上限**（行程条目带线路来源标记；普通加站仍用城市 days 枚举；路书 pickPlan 就近降级兜底），构建器校验每个 days 档 ∈ [站数, Σ城市上限] 可达（2026-07-15 codex F11 轮 F8 复燃后用户拍板）；stops 内不得重复；**线路 alt 取保守口径——任一停留站 alt=true 则线路必须 alt=true**（健康警示不按"主体区域"摊薄）。

数据已备 27 条，重点覆盖 5 天以上时长档，主题化命名（如「河西走廊」「江南水乡」，名称不带日数——F8 后构建器强制）。
