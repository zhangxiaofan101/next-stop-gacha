# Design — 下一站，去哪玩

## 架构（三层，全部封装在单文件页面 + 数据目录中）

```
┌─────────────────────────────────────────────────────┐
│ 展示层  浅色卡通视觉系统 · 筛选控制台 · 卡片网格      │
│         详情页 · 对比表 · 扭蛋机 · 行程单 · 路书视图  │
├─────────────────────────────────────────────────────┤
│ 决策层  过滤/排序引擎 · 条件随机 · 顺路排序(贪心)     │
│         距离→交通方式启发式 · 预算估算 · 路书装配     │
├─────────────────────────────────────────────────────┤
│ 数据层  六区 JSON + 补丁 JSON（仓库 data/）           │
│         构建脚本：合并 → schema 校验 → 注入页面常量   │
└─────────────────────────────────────────────────────┘
```

## 信息组织

| 数据 | 来源 | 更新频率 | 存储 | 用途 |
|------|------|---------|------|------|
| 目的地主数据（144 条） | 模型知识分区生成 + 枚举校验 | 手工低频 | data/六区 JSON | 筛选/展示/随机 |
| 补丁（坐标/住宿/当地交通/体力档/高海拔） | 同上，独立补丁文件 | 手工低频 | data/补丁 JSON | 行程规划/路书/体力筛选 |
| 用户状态（收藏/对比/行程） | 用户操作 | 实时 | localStorage | 跨会话保留 |

## 决策机制

- **过滤**：按属性语义分两类（2026-07-13 用户拍板，判据＝低档位是「主动偏好」还是「被容忍的约束」）
  - **偏好型 → 多选 OR**（选中取并集，选一个不牵连另一个）：地区、季节、冷热、**体力**（躺平/正常/费腿/硬核）——想小众/想硬核是主动选择，不该被自动塞入不想要的档
  - **容忍型 → 天花板**（点一档＝自动含所有更省/更易达的低档，chip 填充式点亮）：**花费**（点档＝预算上限，含更便宜的）、**抵达难度**（点档＝可达性上限，含更好到达的）——高容忍必然接受低，单调
  - 玩法标签为 AND（收窄）；天数为桶映射（2/3/4-5/7/10+ 与目的地 days 数组求交）；关键词全字段模糊
  - 高海拔(>2500m) 为独立二元排除/警示开关，与体力四档正交
  - 实现：偏好型与容忍型共用 `state[key]` Set 与同一条 `!set.has(d.value)` 过滤；差异只在 chip 点击——容忍型点第 t 档填充 {0..t}，故 `.has` 天然得到"≤天花板"
- **空池治理**（M21，2026-07-13 用户拍板，AND 收窄语义保留）：两道防线——①事前：每个未选 chip 实时标注"点下去还剩几个"（按各组真实点击语义模拟），0 的置灰但仍可点；②事后：结果为 0 时计算去掉哪个单一条件（单个玩法 tag / 整组 / 搜索词 / 只看收藏）能救回最多，按救回数给一键定向放宽（保留其余条件），扭蛋空池同样给最优放宽+一键再扭；「全部清空」降级为兜底
- **属性打标原则**（2026-07-13 用户拍板，适用 M14 起所有新属性）：不强制单值——地方有几副面孔就打几档（多值数组），没有明显该属性个性的**留空=通配**（任何该维筛选都命中）；不硬拍主观档位。M16 抵达难度除外（地理事实，天然单值）
- **条件随机**：在当前过滤结果内均匀抽样；动画减速后揭晓
- **顺路排序**：以上海为起点的最近邻贪心，用户可手动覆盖
- **交通启发式**：按坐标球面距离分档（短途打车/自驾 → 中程高铁 → 长程飞机），西藏/新疆/海南/港澳强制倾向飞机+包车；输出明示为估算
- **预算估算**：每站消费档位×天数得住宿餐饮区间，加大交通按距离粗算，输出人均区间
- **路书装配**：按每站所选天数匹配（或就近降配）该站行程方案，串接站间交通段，输出逐日视图 + 纯文本导出 + 打印样式

## 存储与精确性

- 必须精确：schema 枚举字段（构建时强校验，错即拒绝注入）
- 允许漂移：坐标（两位小数，仅用于估算）、住宿/交通建议（知识性文案，页面挂免责）
- 用户输入：无必填输入；全部交互为点选

## 模块列表

### M1 — 目的地数据 schema 与六区数据 [R2 · S3]
144 条核心资产；schema 设计决定所有下游功能的上限，数据错误直接伤害用户信任。

### M2 — 补丁字段（坐标/住宿/当地交通） [R1 · S2]
机械但量大；坐标精度影响交通估算，住宿信息按用户的住宿档位与位置偏好定制（具体品牌偏好不入公开文档，存 assistant 私有记忆）。

### M3 — 构建管线（合并/校验/注入） [R1 · S3]
简单脚本但是质量闸门：枚举、必填、去重、坐标范围全部在此拦截。

### M4 — 筛选/搜索/排序 [R1 · S2]
标准过滤引擎；标签 AND 语义与天数桶映射是仅有的两个非平凡点。

### M5 — 卡片/详情/对比 [R1 · S2]
信息密度与可读性的平衡；对比表复用同一数据投影。

### M6 — 扭蛋随机 [R1 · S1]
核心卖点但实现简单；动画节奏是体验关键。

### M7 — 行程规划器（顺路排序/交通启发/预算） [R2 · S2]
启发式的合理性决定"能不能真用"；估算必须明示，不可伪装精确。

### M8 — 路书生成与导出 [R2 · S2]
逐日装配逻辑 + 文本导出 + 打印 CSS 三种输出的一致性。

### M9 — 视觉系统（浅色卡通） [R2 · S1]
用户明确审美要求；CSP 下无外部字体/图片，卡通感全靠 CSS 造型。

## 二期扩展（2026-07-13 排期，细节见 state.md）

**属性扩展机制**：新属性一律走「schema 增字段 → 补丁文件补数据 → 构建校验收紧枚举 → 前端加筛选组」四步，与既有管线同构；带 🌫️ 的模块先与用户拍板档位定义再动数据。

### M10 — 去过打卡 + 足迹统计 [R1 · S1]
nice-to-have，localStorage 即可。

### M11 — 海外版数据 [R2 · S2]
复用 schema，换区域枚举与出发地假设。

### M12 — 住宿文案去品牌化 [R1 · S2]
公开页面不暴露个人品牌偏好；hotel 字段改档位化+位置化表达，页面模板同步。

### M13 — 花费筛选 [R1 · S1]
cost 字段已存在，纯前端加一组筛选 chips，零数据成本。2026-07-13 语义定为**容忍型·天花板**：点一档自动含所有更省的低档（chip 填充式点亮），过滤 `.has` 不变、仅改 chip 点击填充逻辑。

### M14 — 体力消耗属性 [R2 · S2]
新增 effort 四档（躺平/正常/费腿/硬核）+ 高海拔（>2500m）独立标记（2026-07-13 用户拍板）；按目的地代表性玩法打分，批量补数据。筛选语义＝**偏好型·多选 OR**（选中取并集，不自动联动）；打标＝**多值数组，空=怎么玩都行（通配）**；高海拔为独立二元排除/警示开关。筛选组挂进 M21 的计数/放宽机制。

### M15 — 目的地扩容二批 [R2 · S3]
144 → 218（二批实际 +74，2026-07-13 交付）；复用分区生成+补丁+校验管线，重点补江浙沪县域/海岛、晋豫古建、云贵小众、帕米尔、东北冬季。去重后 74 条为「真实+知名度够+不与既有重复」的可靠上限，未凑数补至 220+。

### M16 — 抵达难度 [R1 · S2]
直达 / 一次中转 / 折腾 三档（2026-07-13 用户确认要做、同日交付），从 transit 半自动派生（启发式：显式"转/换乘/再X"计一跳；未写转字但另有一段限定耗时>30min或坐船/索道/包车进山的落地段也计一跳；近郊自驾算直达，远途飞机后自驾不折算；边境证件/偏远海岛小船/全程约1天等升级为折腾）+ 人工逐条复核全 218 条，影响"周末能不能去"判断。筛选语义＝**容忍型·天花板**（点一档含所有更易达的低档，复用 M13 的 CEIL_GROUPS 机制）。

### M17 — 同行适配 [R1 · S2]
带娃 / 带爸妈 / 独行 / 情侣周末 多选适配（2026-07-13 用户确认要做，2026-07-14 交付）。companions 多值数组存补丁/线路文件，空=谁来都合适（通配），四档全成立时必须留空（构建校验强制 len<4）；筛选＝偏好型多选 OR。与旧「亲子」玩法标签的关系（2026-07-14 用户拍板）：**UI 隐藏、数据保留**——玩法组不再展示亲子 chip，记录数据中的亲子 tag 原样保留（搜索仍可命中，且是带娃打标的强信号）。情侣周末档隐含"短途"（days 含 2/3 且交通不折腾），故长线/线路卡基本无此档。

### M19 — 对比/行程入口显性化 [R1 · S2]
用户反馈（2026-07-13）：对比与行程串联/路书两个核心功能不够显眼。约束：改动不要太夸张，不推翻现有布局。候选手段（实现时择二三）：底部 dock 常驻显示（空态给引导文案与计数 0）、卡片上两个动作按钮加色彩权重、加入第一项后 toast 带「去排行程 →」直达 CTA、首访一次性提示。已先行交付：dock 每篮一键清空 ✕（用户同日追加，免去逐个叉掉）。

### M21 — 空池治理（chip 实时计数 + 定向放宽） [R2 · S2]
解决"很多组合选了直接空空如也"（2026-07-13 用户提出）。机制见「决策机制·空池治理」：matchOne 谓词支持单组条件假设覆盖，chip 计数按真实点击语义模拟，空态/扭蛋按救回数给定向放宽按钮。与 M18 线路卡互补（M18 从供给端扩池，M21 从交互端防空）。

### M18 — 多城联游线路卡 [R2 · S2]
解决"筛 1 周扭蛋空池"（2026-07-13 用户提出，呈现形式拍板：线路卡混入主池）。

**schema（implementer 拍定，非数据层面模糊点）**：独立文件 `data/routes.json`（不改动六区城市文件，零迁移成本），记录字段与城市记录同构（id/name/emoji/province/crowd/cost/seasons/seasonNote/days/transit/tagline/tags/food/museums/architecture/highlights/plans/coords/hotel/local/effort/alt/difficulty 全部必填，直接复用既有校验与筛选逻辑），新增两个字段：
- `regions`: string[]，途经的全部大区（筛选用，多选 OR 命中任一）；`region` 字段保留为主要/起点大区（卡片徽章展示用）。城市记录不填 `regions`，运行时用 `d.regions || [d.region]` 兜底，matchOne 的地区判断因此对两种记录统一处理，城市记录零改动。
- `stops`: `{id, days}[]`，长度 2~4，引用既有城市记录的 id、给出建议天数，即游览顺序。这是"整条装进行程单"的payload——点击即把 stops 逐条 push 进 `state.trip`（遵守既有 6 站上限，超出部分提示部分装入），后续顺路排序/交通估算/路书生成全部复用现有管线，零改动。

`stops` 字段的存在与否是判别"这是线路卡还是城市"的唯一标志（不新增 type 字段，避免侵入城市 schema）。build.py 对含 `stops` 的记录额外校验：长度 2~4、每个 id 必须存在于合并后的城市池、`regions` 非空且 ⊆ REGIONS。

**天数与高海拔契约**（2026-07-14 codex F3/F4 后固化，构建器强制）：站点天数分配是唯一事实真相，卡片/详情展示 Σstop.days 的"全程 N 天"；`days` 枚举只作天数桶筛选档位，须满足 min(days) ≤ Σstop.days ≤ max(days)；stop 建议天数 ∈ [1, 该城市 days 上限]（途经短停允许低于城市独立游枚举，行程下拉将选中值并入选项）；stops 内不得重复；**线路 alt 取保守口径——任一停留站 alt=true 则线路必须 alt=true**（健康警示不按"主体区域"摊薄）。

数据已备 27 条，重点覆盖 5 天以上时长档，主题化命名（如"河西走廊 7 日""江南水乡 5 日"）。
