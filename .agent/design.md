# Design — 下一站，去哪玩

## 架构（三层，全部封装在单文件页面 + 数据目录中）

```
┌─────────────────────────────────────────────────────┐
│ 展示层  浅色卡通视觉系统 · 筛选控制台 · 卡片网格      │
│         详情页 · 对比表 · 扭蛋机 · 行程单 · 路书视图  │
├─────────────────────────────────────────────────────┤
│ 决策层  过滤/排序引擎 · 条件随机 · 顺路排序(贪心)     │
│         距离→交通方式启发式 · 预算估算 · 路书装配     │
├─────────────────────────────────────────────────────┤
│ 数据层  六区 JSON + 补丁 JSON（仓库 data/）           │
│         构建脚本：合并 → schema 校验 → 注入页面常量   │
└─────────────────────────────────────────────────────┘
```

## 信息组织

| 数据 | 来源 | 更新频率 | 存储 | 用途 |
|------|------|---------|------|------|
| 目的地主数据（144 条） | 模型知识分区生成 + 枚举校验 | 手工低频 | data/六区 JSON | 筛选/展示/随机 |
| 补丁（坐标/住宿/当地交通） | 同上，独立补丁文件 | 手工低频 | data/补丁 JSON | 行程规划/路书 |
| 用户状态（收藏/对比/行程） | 用户操作 | 实时 | localStorage | 跨会话保留 |

## 决策机制

- **过滤**：地区/季节/冷热为集合内 OR，玩法标签为 AND（用于收窄），天数为桶映射（2/3/4-5/7/10+ 与目的地 days 数组求交），关键词全字段模糊
- **条件随机**：在当前过滤结果内均匀抽样；动画减速后揭晓
- **顺路排序**：以上海为起点的最近邻贪心，用户可手动覆盖
- **交通启发式**：按坐标球面距离分档（短途打车/自驾 → 中程高铁 → 长程飞机），西藏/新疆/海南/港澳强制倾向飞机+包车；输出明示为估算
- **预算估算**：每站消费档位×天数得住宿餐饮区间，加大交通按距离粗算，输出人均区间
- **路书装配**：按每站所选天数匹配（或就近降配）该站行程方案，串接站间交通段，输出逐日视图 + 纯文本导出 + 打印样式

## 存储与精确性

- 必须精确：schema 枚举字段（构建时强校验，错即拒绝注入）
- 允许漂移：坐标（两位小数，仅用于估算）、住宿/交通建议（知识性文案，页面挂免责）
- 用户输入：无必填输入；全部交互为点选

## 触发总结

| 时机 | 动作 | 状态 |
|------|------|------|
| 数据修改后 | 手动跑构建脚本 | ✅ 可用 |
| 页面加载 | 读 localStorage 恢复收藏/对比/行程 | ✅ 可用 |

## 模块列表

### M1 — 目的地数据 schema 与六区数据 [R2 · S3]
144 条核心资产；schema 设计决定所有下游功能的上限，数据错误直接伤害用户信任。

### M2 — 补丁字段（坐标/住宿/当地交通） [R1 · S2]
机械但量大；坐标精度影响交通估算，住宿信息按用户的住宿档位与位置偏好定制（具体品牌偏好不入公开文档，存 assistant 私有记忆）。

### M3 — 构建管线（合并/校验/注入） [R1 · S3]
简单脚本但是质量闸门：枚举、必填、去重、坐标范围全部在此拦截。

### M4 — 筛选/搜索/排序 [R1 · S2]
标准过滤引擎；标签 AND 语义与天数桶映射是仅有的两个非平凡点。

### M5 — 卡片/详情/对比 [R1 · S2]
信息密度与可读性的平衡；对比表复用同一数据投影。

### M6 — 扭蛋随机 [R1 · S1]
核心卖点但实现简单；动画节奏是体验关键。

### M7 — 行程规划器（顺路排序/交通启发/预算） [R2 · S2]
启发式的合理性决定"能不能真用"；估算必须明示，不可伪装精确。

### M8 — 路书生成与导出 [R2 · S2]
逐日装配逻辑 + 文本导出 + 打印 CSS 三种输出的一致性。

### M9 — 视觉系统（浅色卡通） [R2 · S1]
用户明确审美要求；CSP 下无外部字体/图片，卡通感全靠 CSS 造型。

## 二期扩展（2026-07-13 排期，细节见 state.md）

**属性扩展机制**：新属性一律走「schema 增字段 → 补丁文件补数据 → 构建校验收紧枚举 → 前端加筛选组」四步，与既有管线同构；带 🌫️ 的模块先与用户拍板档位定义再动数据。

### M10 — 去过打卡 + 足迹统计 [R1 · S1]
nice-to-have，localStorage 即可。

### M11 — 海外版数据 [R2 · S2]
复用 schema，换区域枚举与出发地假设。

### M12 — 住宿文案去品牌化 [R1 · S2]
公开页面不暴露个人品牌偏好；hotel 字段改档位化+位置化表达，页面模板同步。

### M13 — 花费筛选 [R1 · S1]
cost 字段已存在，纯前端加一组筛选 chips，零数据成本。

### M14 — 体力消耗属性 [R2 · S2]
新增 effort 四档（躺平/正常/费腿/硬核）+ 高海拔（>2500m）独立标记（2026-07-13 用户拍板）；按目的地代表性玩法打分，批量补数据。

### M15 — 目的地扩容二批 [R2 · S3]
144 → 218（二批实际 +74，2026-07-13 交付）；复用分区生成+补丁+校验管线，重点补江浙沪县域/海岛、晋豫古建、云贵小众、帕米尔、东北冬季。去重后 74 条为「真实+知名度够+不与既有重复」的可靠上限，未凑数补至 220+。

### M16 — 抵达难度 [R1 · S2]
直达 / 一次中转 / 折腾 三档（2026-07-13 用户确认要做），可从 transit 半自动派生，影响"周末能不能去"判断。

### M17 — 同行适配 [R1 · S2]
带娃 / 带爸妈 / 独行 / 情侣周末 多选适配（2026-07-13 用户确认要做），替代单一"亲子"标签。

### M19 — 对比/行程入口显性化 [R1 · S2]
用户反馈（2026-07-13）：对比与行程串联/路书两个核心功能不够显眼。约束：改动不要太夸张，不推翻现有布局。候选手段（实现时择二三）：底部 dock 常驻显示（空态给引导文案与计数 0）、卡片上两个动作按钮加色彩权重、加入第一项后 toast 带「去排行程 →」直达 CTA、首访一次性提示。已先行交付：dock 每篮一键清空 ✕（用户同日追加，免去逐个叉掉）。

### M18 — 多城联游线路卡 [R2 · S2]
解决"筛 1 周扭蛋空池"（2026-07-13 用户提出，呈现形式拍板：线路卡混入主池）。机制：人工精选 2~4 城串线（主题化命名，如"河西走廊 7 日"），独立线路数据文件引用既有城市 id + 每站建议天数；线路卡以"联程票"样式与城市卡同池展示，按总天数参与天数筛选、按途经区域参与地区筛选，同池参与扭蛋；点开可整条装进行程单复用路书管线。数据先备 20~30 条，重点覆盖 5 天以上时长档。
