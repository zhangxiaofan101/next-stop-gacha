## 2. 任务 A — 新城市候选（不强求，宁缺毋滥）

找现有 248 城之外、值得单独成卡的目的地。每条候选给：

| 字段 | 要求 |
|---|---|
| 建议卡名 | 大众目的地语义而非行政区划；同名歧义收窄命名（先例：黄山→黄山风景区、武功山→萍乡·武功山） |
| 省 / 区 | region ∈ {江浙沪, 华东, 华北, 东北, 华中, 华南, 西南, 西北, 港澳} |
| 为何缺 | 对照现有卡说明覆盖空隙（区域 / 玩法 / 季节维度均可） |
| 包含关系检查 | **城市卡之间只能并列**：新卡不得吃掉现有卡的核心内容，也不得被现有卡的行程方案覆盖（M31 原则①④）。逐条写检查结论，点名对照过哪些现有卡 |
| days 建议 | 从 {2,3,4,5,7,10,14} 取 2~3 档 |
| 知名度依据 | 可核查的权威来源（官方文旅 / 景区官网 / 权威媒体），不确定就标注「待核」 |
| P 级 | P1 强烈建议 / P2 可选 |

数量上限 20 条，按 P 级排序。海外不在范围（M11 未启动）。

## 3. 任务 B — 新联程线路候选（重点）

先做**缺口盘点**：把中国大陆（含港澳）出名的自驾环线 / 多城联游线系统过一遍（例：川西大环线、阿里环线、滇西（腾冲瑞丽）、川青甘交界、环塔里木、内蒙草原线、长白山图们江、大兴安岭、湘黔桂交界、闽西土楼线、G219/G331 沿边段……以你的调研为准，不受此例子清单限制），对照现有 37 条逐一判断：已覆盖 / 部分覆盖 / 缺失。盘点表本身就是交付物的一部分。

然后对「缺失」且值得收录的，每条给：

| 字段 | 要求 |
|---|---|
| 建议线路名 | 参照现有命名风格（地理名 + 主题，如「辽西走廊·红滩山海关」） |
| region(s) / province | 枚举同上；跨区线路 regions 可多值 |
| 途经站 | **2~4 站**（硬约束）。每站标注：现有城市卡 id，或「需新建城市卡」（此时该城市同时进任务 A 候选）。长线必须收敛到 ≤4 个停留站——先例：独库全线收敛为 3 站 |
| 站序地理合理性 | 途经顺序必须沿线成立（F19 教训：垭口 / 路段写错方向是 P1 错误）。给出你核对过的地理依据 |
| 天数分配 | 每站单一建议天数 `stop.days`（1 ≤ 且 ≤ 该城市卡 days 上限）；days 档从 {2,3,4,5,7,10,14} 取且必须满足：min(days) ≤ Σstop.days ≤ max(days)，且**每个档 ∈ [站数, Σ各站城市上限]**（build F8 硬校验） |
| alt 判断 | 保守口径：任一停留站 alt=true → 线路必须 true；**风险在路上也算**（翻 3000m+ 垭口 / 达坂即 true，独库先例）。写明依据 |
| 是否需要 leg | stop 指向大区域城市卡、且线路语义是「沿线开」而非「住城放射游」时需要 leg 逐日文案（独库先例）；只是常规联游则不需要。给判断即可，文案由 cc 后写 |
| 上海出发可行性 | 去程大交通（飞 / 高铁到起点枢纽）+ 提还车是否顺路；difficulty ∈ {直达, 一次中转, 折腾} 给个预判 |
| 季节窗口 | 通车期 / 最佳季（seasons ⊆ {春夏秋冬}），有封路 / 限行要点名 |
| 与现有卡冲突检查 | 与 37 条现有线路和 6 张路线型城市卡逐一对照：是否重复、吞并、需迁移 |
| P 级 + 依据 | 同任务 A |

数量上限 15 条，按 P 级排序。

## 4. 红线（违反任何一条 = 白做）

1. **只读调研**：除 `m33-expansion-survey.md` 与 state.md 一行记录外，不改任何文件；不写 data JSON。
2. **杜绝编造**：地名、路况、通车期、海拔全部要有依据；查不实的写「待核」，不要脑补。review 八轴里「地名 / 菜名 / 酒店有无明显编造」也会反过来审你自己的产出。
3. 所有站点 id 引用前先在 `data/` 里确认存在（build 合并全集，可跑 `python3 tools/build.py` 看计数，但不要提交它改动的 index.html——跑完 `git checkout index.html`）。
4. 用户画像供排序参考：上海出发、可自驾（江浙沪短途自驾友好是加分项）、偏好万豪系连锁（沿线有像样酒店的线路加分）、有选择困难（清单要给明确 P 级和推荐理由，别甩 50 条不分主次）。

## 5. 交付格式

`m33-expansion-survey.md` 结构：① 知名线路缺口盘点总表 → ② 线路候选（P1 在前）→ ③ 城市候选（P1 在前）→ ④ 一段「不建议收录」名单及理由（查过但淘汰的，避免下轮重查）。每条候选可直接被用户「要 / 不要」拍板。
