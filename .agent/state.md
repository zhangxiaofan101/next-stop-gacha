  **①的第一版方案已被用户反馈推翻，记录失败原因避免重蹈**：首版用 `mix-blend-mode:multiply`（母版底色本就接近纯白，理论上白色部分该完全透传），现场注入验证后主观判断「无缝融合」即落地为 `[data-theme="porcelain"] .deco-lotus/.deco-cloud/.deco-wave` 规则并 push、部署（Cloudflare 经 wrangler.jsonc 声明的 build.command 随 git push 自动构建，非需要另外手动 deploy 步骤——`getComputedStyle` 实测确认线上 CSS 已生效）。用户特意开私人窗口复验（排除本地缓存变量）后仍反馈「背景还是很白，和周围明显有区别」——real browser zoom 复核确认反馈属实：multiply 只能让**数学纯白**像素透传，三张母版的实际底色是带暖色偏的浅色（非纯白 RGB(255,255,255)），跟页面偏冷的实际底色终究不是同一色，残留的柔和色差在留白大的装饰件上仍清晰可辨——这是 multiply 这招的理论盲区，不是当时验证走了过场。**根治＝母版重出真透明底**：写脚本对每张母版四角采样取背景色，按与背景色的欧氏距离建 alpha（保留柔和渐变边、非硬阈值二值化），并对半透明像素做「反预乘」（un-premultiply：`fg = (pixel − bg·(1−α)) / α`，减去背景残留分量还原前景真实色，避免合成到任意新背景时露色边）——高对比测试背景（粉红）验证零色圈后再定稿。**顺带踩到并解决预算超限**：完整 alpha 通道的信息量远高于纯色底，三张精细线描母版在原 1024px 长边封顶下编码进 60KB 预算失败（`build_illustrations.py` 质量下限 q40 仍超，108~174KB）；反复实测 alpha_q 调低（100→40）只减 ~35%、二值化 alpha 也无实质帮助（信息量瓶颈在轮廓复杂度本身，不在色深/渐变），真正有效的杠杆是分辨率——按各装饰件实际展示尺寸（lotus 150px/cloud 230px/wave 视口内约 600px 上限）核算所需像素密度，母版直接裁到贴近展示尺寸的 2~3 倍（512×512 / 512×267 / 448×224，而非不加区分套用 1024 长边封顶），管线不再降级重编码、零违规过预算（decor-lotus 50.0KB / decor-cloud 41.6KB / decor-wave 54.5KB，均在 60KB 内）。CSS 侧 `mix-blend-mode:multiply` 规则随之整条删除（真透明不再需要该 workaround，留着反而误导后来者）。→ design「装饰位画布契约」条已重写为「不透明母版需 multiply」→ **透明母版是唯一正解，multiply 只在真拿不到透明底时的退路（且已知有色偏残留风险）**。
